# 📅 테스트 실행 계획서 (TEST_PLAN.md)

이 문서는 QA 워크플로우의 2단계인 **'테스트 계획 수립(파이프라인 전략 및 도구 선정)'**의 결과물입니다.
팀원들(개발자, PM, QA)이 모여 "우리 언제, 어떤 툴로, 어떻게 테스트를 돌릴까?"를 회의하고 합의한 규칙(Rule) 문서입니다.

---

## 1. 테스트 도구 선정 (Tooling)
"어떤 망치를 쓸 것인가?"를 정하는 단계입니다.

*   **백엔드 (API & 로직)**: `Pytest`
    *   *선정 이유*: Python 서버(FastAPI)와 가장 궁합이 좋고, 실행 속도가 10초 이내로 매우 빠름.
*   **프론트엔드 & 전체 흐름 (UI/E2E)**: `Playwright`
    *   *선정 이유*: Cypress보다 최신 기술이며, 크롬/사파리 등 다양한 브라우저를 한 번에 띄워서 실제 고객처럼 빠르고 강력하게 테스트할 수 있음.
*   **가상 환경 (모래상자)**: `Docker Compose`
    *   *선정 이유*: 내 컴퓨터, 개발자 컴퓨터, CI 서버 어디서든 똑같은 환경(DB, 서버)을 1초 만에 만들어주기 때문.

---

## 2. 파이프라인 연동 전략 (CI/CD Strategy)
"현장 감독관(GitHub Actions)에게 언제 검사를 시작하라고 명령할 것인가?"를 정하는 단계입니다.

### 🚦 트랙 A: 가벼운 상시 검사 (Push 트리거)
*   **언제?**: 개발자가 코드를 수정해서 GitHub에 올릴(Push) 때마다 즉시 실행
*   **무엇을?**: `Pytest` (백엔드 API 테스트)
*   **이유**: 가볍고 빠르니까(약 1~2분 소요), 개발자가 코드 올릴 때마다 즉시 에러가 났는지 알려주기 위함.

### 🚦 트랙 B: 무거운 전체 검사 (Nightly / PR 트리거)
*   **언제?**: 매일 밤 12시 정각 (또는 실제 운영 서버에 배포하기 직전)
*   **무엇을?**: `Playwright` (프론트엔드 화면을 직접 클릭하는 E2E 테스트)
*   **이유**: 화면을 직접 띄우고 버튼을 누르는 작업은 시간이 오래 걸리므로(무거우므로), 자는 동안이나 아주 중요할 때만 한 번씩 전체적으로 쫙 돌리기 위함.

---

## 3. 누가, 무엇을 준비해야 하는가? (Action Items)
이 계획이 합의되었으므로, 이제 다음 단계(Day 3~4)로 넘어가기 위해 각자 할 일을 나눕니다.

1.  **QA 엔지니어**: 이 규칙에 맞춰 `.github/workflows/qa_automation.yml` (감독관 지시서) 파일을 작성하고, 로봇 코드(`test_login.py` 등)를 짭니다.
2.  **개발자**: QA가 테스트하기 좋게 각 화면의 버튼에 `data-testid="login-btn"` 같은 이름표(Tag)를 달아줍니다.
3.  **인프라 담당자**: 가상 환경이 잘 뜰 수 있도록 `docker-compose.yml` 파일을 세팅해 줍니다.
